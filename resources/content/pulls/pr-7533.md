## Description of the scoring logic used for knowledge query endpoint:

   1. Base Score: The initial ranking from the vector search (ChromaDB) is used as a starting point.
   2. Keyword Matching: It heavily boosts files where keywords from the query appear in important places like the file path, file name,
      or class name. This adds a layer of traditional keyword search on top of the semantic vector search.
   3. Content Type Boosts: guide files get a significant boost, indicating a preference for explanatory documentation over raw source
      code. blog posts get a smaller boost.
   4. Penalties: ticket and release files are penalized (in "all" searches) to prevent them from cluttering the results when you're likely
       looking for implementation details or guides.
   5. Convention Boosts: It boosts Base.mjs files, as these are often foundational classes in the framework.
   6. Inheritance Boost: It cleverly boosts the scores of parent classes in the inheritance chain of a result, acknowledging that context
      is often inherited.
   7. Proximity Boost: It gives a final small boost to files that are in the same directory as other top-ranked files, assuming that
      related files are often located together.

  In short, the algorithm is a multi-faceted approach that combines semantic search with a series of project-specific rules to
  fine-tune the relevance of the results.
